# Internationalization (i18n)

The frontend UI is localizable via [react-i18next](https://react.i18next.com/).
Supported languages: **English (`en`, default/fallback)** and **Portuguese (`pt`)**.

## Where things live

| Concern                | File                                                                   |
| ---------------------- | ---------------------------------------------------------------------- |
| i18n init + helpers    | `frontend/src/i18n/index.ts`                                           |
| Translation strings    | `frontend/src/i18n/locales/{en,pt}.json`                               |
| Language selector (UI) | `frontend/src/components/workspaces/team-settings/LanguageSection.tsx` |
| Per-user persistence   | user config `language` field (backend + `useLanguageSync`)             |

`src/i18n` is imported once in `main.tsx` for its init side-effect.

## How the active language is chosen

In precedence order:

1. **Per-user (UI)** — Team settings → **Idioma / Language**. The choice persists
   in the user config (`language`) and is applied at load by `useLanguageSync`
   (mounted in `WorkspacePage`). Changing it calls `setAppLanguage`, which also
   writes `localStorage` for an instant apply on the next reload.
2. **localStorage** — the i18next `LanguageDetector` restores the last choice
   before the config request resolves (key `kanwas.language`).
3. **Browser `navigator` language** — first-load default for a new device.
4. **Fallback** — `en` when none of the above resolve to a supported language.

## Using translations in a component

```tsx
import { useTranslation } from 'react-i18next'

function Foo() {
  const { t } = useTranslation()
  return <span>{t('explorer.canvases')}</span>
}
```

Outside React (e.g. a TipTap extension config built once), import the instance
directly: `import i18n from '@/i18n'` then `i18n.t('chat.placeholder')`. Note this
reads the language at call time and won't react to a live switch — a reload applies it.

## Adding a string

1. Add the key to **both** `locales/en.json` and `locales/pt.json` (same path).
2. Replace the hardcoded string with `t('namespace.key')`.
3. Keep keys grouped by area (`common`, `explorer`, `settings`, `chat`, …).

## Adding a language

1. Add the code to `SUPPORTED_LANGUAGES` in `src/i18n/index.ts`.
2. Create `locales/<code>.json` and register it in the `resources` map.
3. Extend the backend `language` enum (`user_config_service.ts` +
   `user_config_controller.ts` validator) and the `AppLanguage` usage.
4. Add a `languages.<code>` label to every locale file.

## Scope

Infrastructure, the language selector, and the primary chrome (sidebar/explorer,
team settings, task list, chat composer placeholder) are translated. Remaining
strings are migrated to `t()` incrementally — untranslated text simply renders in
its hardcoded English until a key is added.
